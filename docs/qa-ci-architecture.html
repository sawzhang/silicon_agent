JOB 1 changed-scope
使用 dorny/paths-filter@v3 检测变更范围：
  • platform/** → 输出 backend: true
  • web/** → 输出 frontend: true
后续 lint/test/check jobs 根据此输出条件执行，避免无关代码触发全量 CI。
JOB 2 pr-size-check
统计 PR 中 .py/.ts/.tsx/.js/.jsx 文件的增删行数（排除 package-lock.json 和 uv.lock）。
  • 如果 diff 超过 1000 行，CI 直接失败
  • 强制开发者拆分大型 PR，降低 review 复杂度
  • 仅在 PR 触发时执行（push 不触发）
JOB 3 secret-scan
使用 trufflesecurity/trufflehog@v3.88.0 全量扫描代码库：
  • --only-verified：只报告已验证可用的密钥
  • --results=verified,unknown：包含未知状态的结果
  • 防止 API Key / Token / Secret 意外提交
JOB 4 backend-lint
Python 3.11 环境，pip 缓存基于 pyproject.toml hash。
  • 安装依赖：pip install .[dev]（通过 SSH Agent 拉取私有 skillkit 包）
  • 执行 ruff check app/ tests/
  • 规则：target-version = "py311", line-length = 100
  • 关键规则：F811(重名函数), F401(未用导入), E731(lambda赋值)
JOB 5 backend-test
核心测试 job，覆盖率硬门槛：
  • pytest tests/ -v --tb=short
  • --reruns 2 --reruns-delay 1：flaky 测试自动重试 2 次
  • --cov=app --cov-fail-under=80：低于 80% 直接失败
  • 输出：term-missing + xml + html
  • PR 自动评论覆盖率报告 + 缺失行标注
  • HTML 覆盖率报告上传为 artifact（保留 30 天）
JOB 6 frontend-check
Node.js 20 环境，npm 缓存基于 package-lock.json hash。
  • npm ci：锁定版本安装
  • npx tsc --noEmit：TypeScript strict 模式类型检查
  • npm run build = tsc -b && vite build：完整生产构建
  • 同时验证类型正确性 + 打包完整性
5. CI 优化策略 — 缓存、并发与智能跳过
📦 依赖缓存
Backend: actions/setup-python 的 pip 缓存，key 基于 platform/pyproject.toml 内容 hash

Frontend: actions/setup-node 的 npm 缓存，key 基于 web/package-lock.json 内容 hash

缓存命中时节省 30-60s 安装时间
⚡ 并发控制
concurrency.group: ci-${{ github.ref }}

cancel-in-progress: true

同一分支的新 push 会自动取消正在运行的旧 CI，避免资源浪费（适用于快速迭代的 force-push 场景）
🔍 智能路径过滤
dorny/paths-filter 检测变更目录：

仅 web/ 变更 → 只跑 frontend-check
仅 platform/ 变更 → 只跑 backend-lint + backend-test
两者都变更 → 全量执行

典型节省：50% CI 时间
6. 测试基础设施 — conftest.py 深度解析
platform/tests/conftest.py SESSION SCOPE
环境变量覆写
在导入任何 app 模块前，通过 os.environ 设置测试环境：
WORKER_ENABLED=False · JWT_ENABLED=False · SKILLKIT_ENABLED=False
MEMORY_ENABLED=True · MEMORY_COMPRESSION_ENABLED=False · DEBUG=False
SQLite WAL 模式数据库
使用文件级 SQLite（非内存），路径：sqlite+aiosqlite:////{tempdir}/test_platform_{pid}.db
启用 WAL (Write-Ahead Logging)：读写不互斥，解决并发测试死锁
busy_timeout=30000（30s 写锁等待），NullPool 无连接复用
内存目录隔离
_MEMORY_ROOT 重定向到 tempfile.mkdtemp(prefix="test_memory_")
引擎测试不会写入 platform/memory/ 生产目录
Session & Engine 猴子补丁
app.db.session.engine 和 app.db.session.async_session_factory 被替换为测试专用实例
确保所有 app 内部代码使用测试数据库
setup_db fixture (session-scoped, autouse)
启动时：init_db() 创建全部表
结束后：dispose engine + 清理 *.db / *-wal / *-shm + 移除临时 memory 目录
client fixture
httpx.AsyncClient + ASGITransport 直连 FastAPI app
无需启动真实 HTTP 服务器，测试直接通过 ASGI 协议交互

12. 运行时安全防线 — Circuit Breaker & 多级保护
🛡 Circuit Breaker (熔断器)
触发条件（任一满足即熔断）：
  • 单任务 Token 消耗 > 200,000 tokens
  • 单任务成本 > ¥50 RMB
执行流程：
  • 每个 stage 执行完毕后调用 _check_circuit_breaker()
  • 记录 CircuitBreakerModel（level, status, triggered_by）
  • WebSocket 广播 CB_TRIGGERED 事件到前端
  • 释放 Agent 资源：close_agents_for_task()
  • 标记任务失败：_fail_task()
检查点（4 处）：
  • resume path (L1086)
  • sequential stage (L1152)
  • parallel stage group (L1240)
  • graph-based stage (L1344)
⏱ Timeout 多级超时
任务级超时：
  • WORKER_TASK_TIMEOUT = 1800s (30分钟)
  • asyncio.wait_for(_process_task(), timeout)
  • 超时则 fail task + close agents
Stage 级超时：
  • WORKER_STAGE_TIMEOUT = 300s (默认)
  • 模板中可为每个 stage 单独配置 timeout
🔎 故障分类引擎
worker/failure.py 基于正则匹配：
  • TRANSIENT 超时/网络 → 自动重试
  • TOOL_ERROR 无效 JSON → 文本模式重试
  • RESOURCE 熔断/配额 → 终止
  • SEMANTIC 逻辑错误 → 重试+上下文
  • GATE_REJECTED 人工拒绝 → 终止
🔐 中间件防护栈
  • RequestLoggingMiddleware
    记录 method, path, status, duration + 关联 ID
  • ErrorHandlerMiddleware
    捕获未处理异常 → 500 + 脱敏消息（非 DEBUG 模式）
  • JWTAuthMiddleware
    JWT 认证（JWT_ENABLED=True 时）
    豁免路径: /health, /docs, /ws ...
🔒 原子任务获取
防止多 Worker 竞争同一任务：
  • 步骤1: SELECT 查找 pending 任务
  • 步骤2: UPDATE WHERE status='pending' 原子声明
  • 两步法确保只有一个 Worker 能成功 claim
  • 状态流转: pending → claimed → running
13. Flaky 测试处理与已知问题
🔄 CI Flaky 重试
pytest-rerunfailures 在 CI 中启用：
  • --reruns 2：失败用例最多重试 2 次
  • --reruns-delay 1：重试间隔 1 秒
  • 仅在 CI 环境启用，本地不重试
  • 典型场景：SQLite 并发写冲突、异步时序问题
⚠ 已知 Python 3.13 覆盖率 Bug
coverage.py 7.x + Python 3.13 的 sys.monitoring：
  • 协程 await 恢复后丢失 trace 上下文
  • 表现：await 后的代码行不被追踪（但实际执行）
  • 解决方案：服务单元测试使用 AsyncMock session（同步完成 await）
  • CI 使用 Python 3.11 不受影响
14. 测试编写关键 Gotchas
问题 错误做法 正确做法
sys.modules mock sys.modules["x"] = fake + pop() in finally monkeypatch.setitem(sys.modules, "x", fake) — pytest 自动恢复
重名测试函数 (F811) 同文件两个 def test_foo() 每个测试函数名唯一，第二个会 shadow 第一个（静默跳过）
Lambda 赋值 (E731) f = lambda x: x def f(x): return x
session.refresh Mock 只允许一次调用 Gate 创建调 1 次 + 轮询循环再调，mock 需允许多次
_sort_stages 返回 [] 不提供 stage 对象 至少提供 1 个 dummy stage，否则 _process_task 直接 early-exit
Coverage 异步丢失 真实 DB session 的 await 单元测试用 AsyncMock session 使 await 同步完成
