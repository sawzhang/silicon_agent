2. Agent 创建与池化生命周期
get_agent(role, task_id) 调用链 agents.py
1. 计算缓存 Key
key = f"{role}:{task_id}" — 同一任务同一角色复用 Runner
2. 模型解析
优先级：stage 级 override → LLM_ROLE_MODEL_MAP[role] → 全局 LLM_MODEL
3. 运行时签名比对
对比 (model, temperature, max_tokens, max_turns, skill_dirs, prompt_append)。 签名不变 → 返回缓存实例；签名变化 → 销毁旧实例，创建新 Runner。
4. _create_runner()
构建完整的 SandboxedAgentRunner
创建工作目录 /tmp/silicon_agent/tasks/{task_id}
查找 SYSTEM_PROMPTS[role] (中文角色提示词)
AgentRunner.create(skill_dirs, system_prompt, max_turns)
包装为 SandboxedAgentRunner(engine, config, default_cwd, allowed_tools)
5. 缓存存入
_agents[key] = runner，附加运行时签名用于后续比对。 任务结束调用 close_agents_for_task(task_id) 清理所有该任务的 Runner。
3. 核心 Chat Loop (AgentRunner.chat)
runner.chat(user_prompt, reset=True) skillkit/agent.py reset=True 清空对话历史
INPUT 事件
可拦截/转换输入，支持短路返回（跳过整个 loop）
Skill 检测
检查 /skill-name 模式 → 若匹配，包装 skill 内容为用户消息
追加 User Message 到 _conversation[]
AgentMessage(role="user", content=prompt)
AGENT_START 事件
通知外部系统 agent 开始工作（executor 记录日志）
for turn in range(max_turns):
每一轮 = 一次 LLM 调用 + 可能的多次 Tool 执行
Abort / Steering 检查
协作式取消 + 中途注入用户消息
TURN_START → CONTEXT_TRANSFORM
可修改发送给 LLM 的消息列表
Context Compaction 检查
若 token 估算 > 窗口 × 0.9 → 执行滑动窗口/预算裁剪
_call_llm(messages)
发送 system_prompt + _conversation → OpenAI API
返回 AgentMessage (content + tool_calls + reasoning + usage)
追加 Assistant Message 到 _conversation[]
TURN_END 事件
无 tool_calls → 返回响应 (结束 loop)
纯文本回复 = 最终输出
有 tool_calls → 逐个执行
for each tool_call: BEFORE_TOOL_CALL → execute → AFTER_TOOL_RESULT → 追加 tool result message
回到 loop 顶部，带着 tool results 再次调用 LLM
max_turns 耗尽 → 返回 "Max turns reached" 哨兵消息
Executor 检测后发送 continuation prompts (reset=False)
AGENT_END 事件 (finally)
无论成功/失败/取消，始终触发
4. Tool 执行管线
5 个核心工具
工具参数执行方
executecommand, cwd?BashRuntime.execute()
execute_scriptscript, cwd?BashRuntime.execute_script()
writepath, contentPath.write_text()
readpathPath.read_text() (100K 截断)
skillname, arguments?SkillsEngine (按需加载)
额外：动态 Skill Action 工具 {skill}:{action} + Extension 注册工具
Tool 执行的三层防护
Layer 1 get_tools() 过滤 — 只将角色白名单内的工具声明给 LLM
Layer 2 ToolExecutionPolicyMixin — 运行时拦截：参数校验、cwd 自动注入、白名单二次检查
Layer 3 SandboxedAgentRunner — 路径归一化 + 防逃逸 (_resolve_workspace_path)
降级机制：LLM 返回非法 tool JSON → executor 检测到 tool_call_error → 切换 get_agent_text_only()，禁用 function calling

6. 对话管理 (_conversation)
AgentMessage 结构
字段类型说明
rolestr"user" | "assistant" | "system" | "tool"
contentstr | list文本或多模态内容 (TextContent | ImageContent)
tool_callslist[dict][{id, name, arguments}, ...] — LLM 请求的工具调用
tool_call_idstr?tool result 消息的关联 ID
reasoningstr?MiniMax reasoning_split 推理内容
token_usageTokenUsage?本轮 input/output/cache/thinking tokens
metadatadictfinish_reason, model 等
Conversation 生命周期
reset=True 清空 _conversation[]，全新上下文
reset=False 保留历史，继续对话（用于 continuation / evaluator loop）
steer(msg) 注入用户消息到 steering_queue，下一轮前插入
follow_up(msg) 当前 loop 结束后自动发送后续消息
abort() 协作式取消，在 tool 执行间隙检查
关键设计：跨 stage 调用始终 reset=True — Agent 不共享对话历史， 所有上下文靠 build_user_prompt() 显式注入
7. System Prompt 三层组装
AgentRunner.build_system_prompt()
1
SYSTEM_PROMPTS[role] — 角色身份定义（中文）+ 工作目录提示 + system_prompt_append
2
Context Files — AGENTS.md / CLAUDE.md (若 load_context_files=True，当前关闭)
3
Skills Prompt — 已加载的 Skill 描述（XML 格式，16000 字符预算）+ 可调用技能列表
System Prompt 在 Runner 创建时固定，整个任务期间不变。动态内容（prior outputs、retry context 等）通过 User Message 注入。

9. 上下文窗口管理 (ContextManager)
Token 估算
estimate_tokens(text)
字符数 / 4 的启发式估算。每条消息额外 +4 tokens（角色标记开销）。 计入 content + reasoning + tool_call arguments。
压缩触发
should_compact(messages)
当 token 估算 > context_window × 0.9 时触发。 默认 context_window = 128K，reserve_tokens = 4096。
两种压缩策略
SlidingWindow 保留最近 N 轮，超出预算从前向后裁剪
TokenBudget 从最新消息反向累积，直到填满预算
约束：裁剪后首条消息必须为 user 或 system 角色
10. LLM 调用路径 (_call_llm)
_call_llm(messages) 双路径分发
路径 A：内置 OpenAI Client 当前使用
1. 懒初始化 AsyncOpenAI (httpx, trust_env=False, 300s timeout)
2. 格式化 messages → OpenAI API 格式
3. 构建 kwargs: model, messages, temperature, max_tokens, tools
4. MiniMax 特殊: extra_body.reasoning_split=True
5. client.chat.completions.create(**kwargs)
6. 提取 content, reasoning, tool_calls, usage → AgentMessage
路径 B：LLMAdapter 可扩展
1. 检查 self.active_adapter (AdapterRegistry)
2. 若不为 None → adapter.chat(messages, system_prompt)
3. 返回 AgentResponse → 转换为 AgentMessage
4. 支持自定义 Provider (Anthropic, 本地模型等)

当前 AdapterRegistry 为空，所有调用走路径 A
11. Skills 系统
Skill 加载与管理 (SkillsEngine)
加载：扫描 skill_dirs 下的 SKILL.md 文件 (YAML frontmatter + Markdown body)
过滤：DefaultSkillFilter — 平台兼容性、环境变量、依赖检查
缓存：get_snapshot() 快照缓存，文件变更时失效
格式：Skills 以 XML 格式注入 System Prompt（16K 字符预算）
热加载：watchfiles 监听 skill 目录变更
两种调用方式
Tool 调用 LLM 发起 skill(name="xxx", arguments="...") → 返回 skill 内容作为 tool result → LLM 据此继续工作
Slash 命令 用户输入 /skill-name args → skill 内容包装为 <skill-content> 标签注入用户消息
Fork 模式：标记 context: fork 的 skill 会创建子 AgentRunner， 用 skill 内容作为 system prompt，独立执行后返回结果。
动态注入：Skill 中的 !`command` 模式会在加载时执行，stdout 替换占位符。
12. Token 消耗追踪与熔断
TokenUsage 五维追踪
维度来源
input_tokensprompt_tokens (含 cache_read)
output_tokenscompletion_tokens
cache_read_tokensprompt_tokens_details.cached_tokens
cache_write_tokens自动计算 (input - cache_read)
thinking_tokensreasoning output tokens
支持 + / += 运算符累积。 cumulative_usage 记录整个 Runner 生命周期的总量。 reset_usage() 在每个 stage 开始前归零。
熔断器 (Circuit Breaker)
Token 限制 CB_MAX_TOKENS_PER_TASK = 200K — 单任务总 token 上限
成本限制 CB_MAX_COST_PER_TASK_RMB = ¥50 — 单任务成本上限
Stage 超时 WORKER_STAGE_TIMEOUT = 300s — 单次 LLM 调用超时
任务超时 WORKER_TASK_TIMEOUT = 1800s — 整个任务总超时
Executor 每轮 tool 执行后检查 cumulative_usage.total_tokens， 超限则终止 stage 并标记 failed。
13. Executor 容错与续写机制
重试策略
最多 WORKER_STAGE_MAX_RETRIES + 1 次尝试。 指数退避：WORKER_STAGE_RETRY_DELAY × 2^attempt。 超时/API 错误/未知错误均可重试。
Tool Fallback
检测 LLM 返回非法 tool JSON → get_agent_text_only() 禁用 function calling → 纯文本模式重试，缓存键带 :textonly 后缀。
Continuation 续写
输出包含 "Max turns reached" → 发送 "请继续" prompt (reset=False) → 最多 3 次续写 → 合并所有输出。
