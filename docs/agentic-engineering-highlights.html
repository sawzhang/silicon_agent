Part I — Pipeline 编排层
1. 声明式模板 + 双模式执行引擎
同一 JSON 模板自动适配线性/DAG 两种执行模式
TaskTemplateModel.stages AUTO-DETECT
Template JSON 定义 stages 数组
每个 stage 包含：name, agent_role, order（线性模式）, depends_on（DAG 模式）
可选字段：model（模型覆盖）, instruction, timeout, condition, on_failure, max_executions, evaluator, context_from
Engine 检测执行模式
扫描 stages 数组 — 若任一 stage 含 depends_on 字段 → DAG 模式 → StageGraph
否则 → 线性模式 → 按 order 值排序，同 order 并行执行
向后兼容
旧模板无 depends_on → 自动走线性模式，无需任何迁移
新模板可混用：部分 stage 有 depends_on，引擎自动补全缺失的依赖关系

🔄 Ready-Set 算法
get_ready_stages() 判定条件：
  • 所有依赖 stage 已 completed 或 skipped
  • 自身不在 running 状态
  • 执行次数 < max_executions
  • 失败 stage 且未达上限 → 可重新调度
💥 失败重定向
Stage 定义 on_failure: "target_stage"：
  • test 失败 → 重置 code 为 pending
  • 从 completed/failed 集合中移除 target
  • 下一轮循环自动重新执行
  • 配合 max_executions 防止无限重跑
✅ 安全验证
validate() 方法：
  • DFS 三色法（WHITE/GRAY/BLACK）环检测
  • 缺失依赖项检测
  • 全局循环上限：GRAPH_MAX_LOOP_ITERATIONS × len(nodes)

4. 多模型路由 — 3 级决议体系
agents.py resolve_model_for_role — Stage > Role > Global 三级覆盖
优先级 1
Stage 级
stage_defs[name]["model"]
模板中每个 stage 单独指定模型
例: test stage 用便宜快速模型
优先级 2
Role 级
LLM_ROLE_MODEL_MAP
JSON 映射: {"coding": "gpt-4o", "review": "claude-sonnet"}

优先级 3
Global 级
LLM_MODEL
全局默认模型（兜底）
所有未指定模型的 stage/role 使用此模型
Runtime Signature 缓存机制
Agent Runner 按 (role, task_id) 缓存复用。每次 stage 执行时计算当前 config 的签名元组（model, temperature, max_tokens, ...）， 若签名与缓存中不一致 → 自动销毁旧 Runner 并重建。适用于同一 role 在重试时切换到不同模型的场景。
Part II — Human-in-the-Loop 层
5. Gate 四种人机协作模式
从被动审批到主动求助 — 全维度的人类介入机制
MODE A Static Gate — 模板定义
在模板 gates JSON 中声明：
{"after_stage": "spec", "type": "human_approve", "max_retries": 2}
  • Stage 完成后自动创建 HumanGateModel
  • Worker 每 3 秒轮询 DB 直到 approved/rejected
  • 通过 WebSocket 广播 gate:created
  • 外部通知（飞书/钉钉）同步推送
MODE B Dynamic Gate — 信心驱动
DYNAMIC_GATE_ENABLED=True +
DYNAMIC_GATE_CONFIDENCE_THRESHOLD=0.5
  • 每个 stage 完成后检查 output_structured.confidence
  • 低于阈值 → 自动插入 confidence_review gate
  • 标记 is_dynamic=True 区分于模板 gate
  • Agent 主动"承认"不确信 → 请求人类帮助
这是最有亮点的设计 — 实现了 Agent 的自我认知边界感知
MODE C Rejection Feedback — 反馈循环
Gate 被拒绝 + current_retry < max_retries：
  • 构造 gate_rejection_context（审阅者评论 + 重试次数）
  • 重置 stage 为 pending，清空 output/error
  • 重跑 stage 时注入 prompt：
    "## Gate 审批被拒绝 — 请根据反馈修改"
  • Agent 看到人类的具体修改意见后重新执行
MODE D Revise & Continue — 修订继续
第三种 Gate 决议（非 approve 非 reject）：
  • 审阅者提供 revised_content 修改稿
  • 不完全拒绝，而是携带修订稿继续执行后续 stage
  • 适用场景：输出大方向正确但需微调的情况
  • 减少不必要的全量重跑开销

Part III — 自进化层
7. Self-Evaluation Loop — 自批判迭代精炼
executor.py _run_evaluator_loop_inner — Agent 自评 + 循环改进直到达标
evaluator: {"enabled": true, "criteria": "...", "min_confidence": 0.7} MAX 3 ITERATIONS
Step 1: 初始执行
Agent 按正常流程执行 stage，生成初始输出
Step 2: 自我评估
Agent 对自己的输出生成结构化评估：
{"confidence": 0.x, "issues": ["问题1", "问题2"], "suggestions": ["建议1"]}
Step 3: 判断 confidence ≥ min_confidence ?
YES → 通过，进入下一 stage
NO → 将 issues + suggestions 注入 prompt，回到 Step 1 重新执行
Step 4: 结果记录
最终 self_assessment_score 存储在 stage 上 → 可作为 Dynamic Gate 判断依据
最多 max_iterations 次（默认 3），超过强制通过

8. Ralph Loop V2 — 结构化失败反思 + 智能重试
worker/failure.py — 6 类故障分类 + LLM 反思三元组 + 跨任务经验沉淀
💥 6 类故障分类引擎
基于正则匹配错误消息，自动分类并决定处理策略：
类别触发模式处理
TRANSIENT timeout, network, rate_limit 自动重试
TOOL_ERROR invalid JSON, tool failure 文本模式重试
enable_tools=False
RESOURCE circuit breaker, quota 终止
SEMANTIC wrong output, quality issue 增强上下文重试
GATE_REJECTED 人工拒绝 反馈循环重试
UNKNOWN 无法分类 标记失败
🧠 结构化反思三元组
SKILL_REFLECTION_ENABLED=True 时，失败后 LLM 分析错误生成：
{ "root_cause": "SQL查询缺少JOIN导致字段不存在", "lesson": "关联查询时必须显式JOIN所有引用的表", "suggestion": "在SELECT前添加LEFT JOIN user_profiles" }
三元组去向：
  • root_cause + lesson + suggestion 全部注入重试 prompt
  • lesson 持久化到 Project Memory
    （issues 类别，confidence=0.7）
  • 效果：A 任务踩的坑，B 任务天然避开
